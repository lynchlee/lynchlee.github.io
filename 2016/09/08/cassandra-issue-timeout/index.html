Cassandra Issue - Timeout while setting keyspace on connection

Timeout during the first connection:

After after the reconnect, works. :/

Reason:

this happens while setting the keyspace on a new connection: if your session is tied to a particular keyspace (either by using cluster.connect(String) or issuing a USE statement later), any time a new connection is opened, the first request to use this connection will first send a “USE ks” query to switch to the keyspace.
What you see here is that “USE ks” statement timing out, i.e. the server takes longer than SocketOptions#getReadTimeoutMillis to reply. That’s surprising because a USE statement is normally very fast to execute, so it could indicate a server or network issue. Do you see anything in the Cassandra logs?

More, While faced this type of issue, maybe, we should think about:

Have we customized any of the following:

  • SocketOptions.setReadTimeoutMillis() on the driver side?
  • *_timeout_in_ms in cassandra.yaml?
  • Do us have statements that use IN queries with a large number of partition keys?
